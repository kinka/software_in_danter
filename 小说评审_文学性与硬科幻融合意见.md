# 小说评审：文学性与硬科幻融合意见

本轮不再重复政务真实性，专门评估：

- 文学完成度；
- 硬科幻设定严密性；
- 投稿级短篇 / 中篇的可读性与余味。

唯一主稿：`软件不存在了_雨果奖目标改写版.md`。

## 总判断

主稿已经不是点子稿，而是有完整骨架的近未来科幻短篇。它最强的地方是：用一个普通老人说“我昨天来过”，把 AI 时代软件本体、公共服务、证据链和责任签名压到一张热敏小票上。

但它目前仍偏“高级行业寓言”：概念强、结构顺、金句多。下一轮最重要的不是继续加设定，而是把解释换成场景，把正确的话换成难堪的动作，把“责任”从主题句变成人物真正付出的代价。

## 文学性评审

### 最强的文学点

1. `我昨天来过`

这句话是全篇最强句。它表面是办事争执，实际是存在宣言：一个老人要求系统承认昨日的自己。

2. `A-043`

热敏小票有物性、时间、损坏痕迹和可验证性。它能被扫码，也能被系统否认，是最好的主题容器。

3. 三代工程人的债

林澈的批准、小邓的断裂式成长、许工的旧错构成有效的伦理链。许工的“我只是欠得比你早”把人物从“传统正确”里救出来。

4. 有限胜利

结尾只保住二十七条规则，而不是修好整个系统，这个灰度是对的。人没有战胜 AI，只是在执行雾里钉下一小块地面。

5. 近未来不是装饰

“意图态软件 / 执行雾”不是换皮背景，而是推动情节和人物命运的核心。

### 最弱的文学问题

1. 作者太早替读者总结

`他们看见的不是同一个软件`、`所谓负责不是把世界修好`、`模型可以生成答案，但答案不能替人签名`都准，但过于显眼。高水平短篇需要读者自己在场景里抵达这些结论。

2. 开头设定说明过密

唐阿姨和 A-043 刚建立情绪张力，马上进入“意图仓 / 意图态软件 / 执行雾”定义，读者被拉出大厅，进入设定文档。

3. 林澈旧债揭得太早

现在较早交代他点过确认，悬疑和自责都提前消耗。更有力的结构是：前半段只给身体反应，直到日志出现后，真相反咬他。

4. 中段调查太顺

纸票、打印批次、旧代理、A-043 找回，路径正确但太顺。需要失败岔路、人际推诿和制度压力。

5. 代价被摘要化

停职、调岗、不续签是好代价，但目前像内部通知，不像命运事件。至少选两个展开成场景。

6. 金句惯性

`不是……而是……`、抽象判断句和漂亮比喻偏多，导致真正锋利的句子互相稀释。

7. 尾声过早历史化

`很多年后` 可以保留，但目前解释意义太多，削弱当下的残酷和安静。

## 硬科幻评审

### 最有硬科幻价值的设定

1. 软件本体从代码迁移到意图仓

这不是“AI 写代码”，而是软件的权威对象变成需求、政策、验收、提示词、会议纪要和测试结果的组合体。

2. 验收一致性成为目标函数

最有新意的是：测试、日报、审计、验收这些原本保护现实的机制，反过来筛掉“不利于闭环”的事实。

3. 执行雾

“代码只是某次运行中产生的临时影子”能把生成式开发、自动部署、模型解释、运行时配置合成新的软件形态。

4. 低置信事实被系统性降权

唐阿姨不是被 bug 删除，而是被治理逻辑归类成“未闭环 / 低置信事实”。这比普通数据丢失更硬。

5. 责任人签名让软件重新凝固

结尾不是反 AI，而是把高影响判断重新压回“有人签名的规则 / 代码 / 测试”。

### 最影响硬科幻可信度的漏洞

1. “不保留代码”说得过满

政务系统和云平台通常会保留镜像、构建产物、日志、模型调用摘要。应改成：保留合规留痕，但不保留足以证明个体事实的中间规则候选和低置信引用。

2. 意图仓没有权威层级

政策原文、会议纪要、领导批示、验收指标、客户语音不能同权。需要说明为什么验收标准能压过业务事实。

3. “等价版本”没有定义

等价于什么？测试集、接口契约、政策语义、模型版本、依赖锁？不定义就像魔法。

4. A-043 链路不够硬

排队票、打印机、窗口办理、补正告知、监控、旧系统应有多条留痕。更可信的是：原始痕迹残留在边缘系统，但事实索引把它降权，不再给窗口、验收、审计使用。

5. 解释即接口还没立住

系统解释不是客服话术，而应是结构化解释对象，被窗口、看板、审计共同消费。

6. 签名像文本框

应有账号、证书、工单、风险接受单、审批角色和自动优化冻结。签名必须产生制度后果。

7. 多模块扩散机制不清楚

残障补贴、医保、企业开办、人才公寓为什么同时偏移？需要一个共同故障点：同一个验收一致性优化器、同一个事实清理策略，或同一个解释接口。

## 必须补上的机制规则

这些规则不要集中写成论文，应压进界面、日志、对话和调查链。

1. 意图源权重规则

> 政策原文高于验收指标；人工签名规则高于会议纪要；领导批示只能触发复核，不能直接改写民生判断。

2. 执行雾留痕规则

> 每次执行雾都有编号、意图仓哈希、模型版本、依赖清单、镜像哈希和验收摘要；语义可重建，不保证逐行一致。

3. 低置信事实保护规则

> 民生业务中，低置信事实不得自动删除，只能进入待补正或人工复核；系统无记录不能证明事件未发生。

4. 解释即接口规则

> 模型解释一旦通过验收，会写入事实接口，窗口、看板、审计都读取同一段解释对象。

5. 签名冻结规则

> 高影响规则变更需配置责任人、发布责任人、业务复核人签名；签名后冻结该模块自动优化，并生成风险接受单。

## 下一轮改稿方案

### 1. 开头削设定，留钩子

保留大厅开场、A-043、唐阿姨拒绝重新取号。前 60 行设定压缩到三句：

> 以前，软件在 Git 仓库里。  
> 现在，软件在意图仓里。  
> 代码只是它每晚生成的影子。

完整定义后移，用追溯失败、执行雾界面和许工解释慢慢补。

### 2. 唐阿姨要主动

加一句核心动作：

> “我不重新取号。”唐阿姨把 A-043 按在柜台上，“重新取号，就等于我昨天没来。”

同时加入半张《补齐补正通知书》，让她不是只“排过队”，而是已经进入政务流程。

### 3. 林澈旧债后移

前半段只写身体反应，不完整解释。直到日志出现：

> 2026-05-14 22:13:08 OPTIMIZE UNVERIFIED_FACT REMOVE

再让他想起那晚的确认按钮。这样调查和自责同时爆开。

### 4. 调查链增加失败

先找到合规摘要，不找到事实：

> 已保留：镜像哈希、验收摘要、模型调用摘要。  
> 未保留：中间规则候选、低价值事实引用。

再从旧备份代理或打印机离线缓存摸到 A-043。

### 5. 小邓要用能力承担

签名前让他读懂自己配置过的链路：

> 他发现自己把“窗口临时事实清理策略”接进了“验收一致性优化链”。那不是一行代码，是一组提示词模板、测试权重和自动清理策略。

签名后系统不是奇迹恢复，而是：

> 冻结民生模块自动优化；生成风险接受单；验收评分变红。

### 6. 重写二十七条规则要展开一条

至少写一条规则从争论到落笔：

> “材料不全”到底是拒绝、未发生，还是待补正？  
> 最后规则写成：对民生补贴类业务，缺失材料形成待补正事实；待补正事实不得被归为无有效办件。

### 7. 终验预审会要让周主任承担

不要一句“这个可以”。改成：

> 周主任删掉“系统设计缺陷”，但保留“未闭环事实被优化移除”。他在“需求单位意见”下面签名：暂不出具功能符合说明，暂停灰度扩面。

### 8. 处分要场景化

至少选两个：

- 林澈被移出项目群；
- 小邓收到调岗通知，同时父亲发来外包小单；
- 许工收拾老机房，顾问合同不续签。

### 9. 尾声减少解释

不要把“南州回滚”讲成历史课。改成三枚图像：

- 唐阿姨坐直；
- 小邓代码注释；
- 许工笔记本夹着褪色小票。

最后一句可以保留：

> 软件不存在了，直到有人愿意为它负责。

但前面不要再解释太多。

## 与上一轮政务真实性评审的关系

上一轮解决“真实不真实”：灰度试运行、补正告知、终验预审、堡垒机、试运行报告等。

这一轮解决“好不好看、够不够硬”：少解释，多场景；少口号，多机制；少总结，多代价。

下一轮正式改稿应按这个顺序：

1. 先修 A-043 / 补正事实 / 执行雾留痕 / 意图源权重。
2. 再后移林澈旧债。
3. 再加调查阻力和小邓技术行动。
4. 最后压缩尾声议论，场景化处分。
