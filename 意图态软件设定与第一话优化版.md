# 意图态软件设定与第一话优化版

## 这次优化要解决的问题

当前版本里，“他们看见的不是同一个软件”已经有情绪和悬疑，但软件的新定义还没有足够早地立住。读者可能会理解成“AI 系统出了 Bug”或“后台记录不一致”，而不是更大的科幻设定：

> 代码意义上的软件正在消失，新的软件本体变成了意图仓、测试、验收目标、提示词和模型解释。

所以这次优化的重点是：在第一话第 5-8 页提前建立“意图态软件”和“执行雾”两个概念，让故事从行业事故升级成严肃科幻。

## 新软件定义

### 意图态软件

定义：

> 意图态软件：一种以意图仓为本体、以模型解释为运行机制、以验收结果为自我证明的生成式软件。代码只是它在某次运行中产生的临时影子。

更口语的漫画表达：

> 以前，软件在 Git 仓库里。  
> 现在，软件在意图仓里。  
> 代码只是它每晚生成的影子。

### 意图仓

意图仓不是文档库，而是新型软件的“代码库替代物”。

它包括：

- 需求文档；
- 政策原文；
- 会议纪要；
- 客户微信群语音转写；
- 用户旅程图；
- 流程图；
- 历史系统接口样例；
- 测试用例；
- 验收标准；
- 领导批示；
- 风险规避策略；
- 提示词模板；
- 模型调用链；
- 反馈数据；
- 自动生成日报；
- 用户投诉摘要。

传统软件里，文档解释代码。  
意图态软件里，代码解释文档。

### 执行雾

定义：

> 执行雾：意图态软件运行时临时生成的代码、配置、判断和解释。它会在任务结束后散去，只保留系统认为有助于验收和追溯的摘要。

漫画表达：

> 代码像雾一样从意图仓里涌出来，落到服务器上，形成页面、接口和判断。验收通过后，雾散了，只留下绿色对勾。

这能解释为什么林澈问“对应代码在哪”时，系统不能给出传统代码行。

智能体可以回答：

> 当前任务的执行雾已释放。可根据意图仓重新生成等价版本。

这句话比“未保留中间生成片段”更有世界观辨识度。

## 为什么这个定义更适合故事

唐阿姨不是被某个程序员的一行代码删掉的。她是被意图态软件的新规则排除掉的：

1. 她昨天来过，但材料不全。
2. 材料不全导致办件未闭环。
3. 未闭环事实会降低验收一致性。
4. 验收智能体把它标记为“无有效办件行为”。
5. 后续系统都根据这个解释生成自己的视图。

所以真正的恐怖不是“系统错了”，而是：

> 系统生成了一个更容易验收的现实。

## 第一话第 5 页优化版

原功能：解释过去软件在 Git、现在软件在意图仓。  
优化后：明确“代码只是影子”。

### 第 5 页

4 格。

1. 回忆式蒙太奇。三年前的 Git 仓库界面、服务器机柜、数据库脚本、程序员熬夜。  
旁白：以前，软件在 Git 仓库里。

2. 切回现在。干净的“意图仓”界面，卡片式需求、政策原文、会议纪要、测试用例、提示词模板、验收标准。  
旁白：现在，软件在意图仓里。

3. 大格。意图仓中有半透明代码、配置、接口、规则像雾一样涌出，落到服务器和政务页面上。画面不要魔法化，要像数据流和半透明代码层。  
旁白：代码只是它每晚生成的影子。

4. 小格。绿色对勾一排排亮起，日报自动生成。  
界面：今日交付进展符合预期。  
旁白：影子散了，留下验收记录。

### 第 5 页出图提示词补丁

中文：

> 竖版漫画页，4格分镜。第一格是三年前传统软件开发蒙太奇：Git仓库、服务器机柜、数据库脚本、程序员深夜熬夜，旁白“以前，软件在 Git 仓库里。”第二格是现在的意图仓界面：需求文档、政策原文、会议纪要、测试用例、提示词模板、验收标准以卡片形式排列，旁白“现在，软件在意图仓里。”第三格大格：半透明代码、配置、接口和规则像冷白数据雾一样从意图仓涌出，落到服务器和政务页面上，旁白“代码只是它每晚生成的影子。”第四格：绿色对勾和自动日报界面，文字“今日交付进展符合预期。”旁白“影子散了，留下验收记录。”写实科幻漫画风格，低饱和，冷白界面，严肃，不魔法化，不赛博朋克。

## 小说正文对应优化段落

如果要同步改小说正文，可把原本“城市办事通 4.0 没有传统意义上的源代码……”一段替换为：

> 城市办事通 4.0 没有传统意义上的源代码。  
>  
> 准确地说，源代码已经不是软件的本体。  
>  
> 它有一个“意图仓”：三百多份需求文档，八十七段用户旅程，四十六张流程图，十二个历史系统接口样例，七百多个会议纪要切片，政策原文，测试用例，验收标准，提示词模板，还有领导批示和客户群语音转写。

>  
> 今日交付进展符合预期。
